    API 인증키 발급 + 청약 3종 API 구조

    인증키를 받고, 분양·경쟁률·당첨 3개 API를 연결하는 법

    인증키 발급 4단계 인증키 형태 청약 3종 API 공통 연결 키

    2-1인증키 발급 4단계 + 인증키 형태

    API를 부르려면 ‘출입증(인증키)’이 필요하다

  • 데이터 검색 — “아파트 분양정보” 등 검색
  • 활용 신청 — 활용신청 버튼 ▸ 동의
  • 승인 확인 — 보통 즉시(마이페이지)
  • 인증키 복사 — 일반/Encoding 키 확인
    • 인증키 형태 (서비스키)# 일반 인증키 (Decoding) aBcD1234eFgH5678...%2F... # 길고 특수문자 포함 # Encoding 인증키 — URL에 그대로 붙일 때 사용 aBcD1234eFgH5678...%252F...
    주의 — 인증키는 비밀번호. 코드에 그대로 적거나 GitHub에 올리면 안 된다(섹션 3 .env). 인코딩/디코딩 키 구분 주의(에러 시 바꿔 시도).

    2-2청약 3단계 × 3개 API

    분양의 흐름(분양 → 경쟁 → 당첨)이 그대로 3개 API

    단계API데이터 ID내용
    ① 분양분양정보15098547단지·일정·공급·분양가
    ② 경쟁경쟁률15098905주택형별 청약 경쟁률
    ③ 당첨당첨자15110812당첨 가점·커트라인
    스토리 — “이 단지가(분양정보) 얼마나 인기였고(경쟁률) 누가 됐나(당첨자)”. 세 API를 합치면 청약 전 과정의 인사이트가 나온다.

    2-3공통 연결 키

    세 API를 ‘하나의 단지’로 잇는 공통 식별자

    🔑공통 키

    ‘주택관리번호’ 같은 공통 식별자로 분양정보↔경쟁률↔당첨을 연결

    🧩왜 필요

    같은 단지의 데이터를 합쳐서 보여주려면 ‘무엇이 같은 단지인지’ 기준이 필요

    연결 개념 (의사 코드)# 주택관리번호로 3개 API 데이터 합치기 분양 = 분양정보API("주택관리번호") 경쟁 = 경쟁률API(분양."주택관리번호") 당첨 = 당첨자API(분양."주택관리번호") 대시보드 = 합치기(분양, 경쟁, 당첨)
    섹션 2 정리 — 인증키로 출입하고, 3종 API를 공통 키로 연결. 이제 ‘어떻게 호출하는지(REST)’와 ‘키를 안전하게 두는 법’을 배운다.

    03

    REST API 호출 구조 + 인증키 보안(.env)

    API를 부르는 표준 방식과, 키를 안전하게 숨기는 법

    REST API란 호출 구조 인증키 보안(.env)

    3-1REST API란 + 호출 구조

    ‘주소(URL)로 데이터를 요청’하는 표준 방식

    📡REST의 기본

    • URL로 요청 대상 지정
    • GET(조회)이 대부분
    • 파라미터로 조건 전달
    • 응답은 JSON/XML

    🧱구성 요소

    • 엔드포인트(주소)
    • serviceKey(인증)
    • 쿼리(지역·기간 등)
    청약 분양정보 API 호출 (구조)GET https://api.odcloud.kr/api/15098547/v1/uddi:... ?serviceKey=발급키 &page=1 &perPage=100 &cond[지역명::EQ]=서울 # 응답(JSON): { "data": [ { 단지명, 공급규모, 청약일정 ... } ] }
    가이드 문서 — 각 API는 ‘활용가이드(명세서)’를 제공한다. 막히면 명세서 PDF를 Claude에 붙여 “이 API 호출 코드 만들어줘”로 해결.

    3-2인증키 보안 — .env

    키를 코드에 직접 적지 않고 ‘비밀 파일’에 숨긴다

    .env (비밀 파일 — 절대 공개 금지)# 키는 여기에만 DATA_GO_KR_KEY=aBcD1234eFgH5678... # 코드에서는 이름으로 불러옴 (값은 안 보임) const key = process.env.DATA_GO_KR_KEY;
    .gitignore (이 파일을 깃에서 제외)# .env를 커밋·업로드에서 빼기 .env .env.local
    철칙 — 인증키를 코드에 직접 쓰면 GitHub에 ‘그대로 노출’된다. .env에 넣고 .gitignore로 제외. 유출 시 즉시 재발급.
    실제 적용 — 이 강의 사이트도 민감정보는 커밋 제외(.gitignore) 원칙을 지켰다. 보안은 ‘습관’.

    GitHub 개념 + GitHub Pages·CLI 배포

    코드를 저장·공유하고, 무료로 인터넷에 올리기

    Git·GitHub 개념 GitHub Pages 배포 GitHub CLI(gh)

    5-1Git · GitHub 개념

    ‘버전 저장’과 ‘클라우드 공유’

    💾Git

    코드의 버전을 저장하는 도구. ‘언제·무엇을 바꿨는지’ 기록(커밋)

    ☁️GitHub

    Git 저장소를 클라우드에 보관·공유. 협업·배포의 중심

    비유 — Git = ‘구글 문서의 버전 기록’, GitHub = ‘그 문서를 올려두는 구글 드라이브’. commit(저장) → push(업로드)가 기본 흐름.
    기본 흐름 (이 사이트도 이렇게)git add . # 변경 담기 git commit -m "메시지" # 저장(스냅샷) git push # GitHub에 업로드

    5-2GitHub Pages 배포

    정적 사이트(HTML)를 ‘무료로’ 인터넷에 — 이 강의자료 사이트가 그 예

  • GitHub에 저장소(repo) 생성·push
  • Settings ▸ Pages ▸ 브랜치(main)·폴더(/) 지정
  • 잠시 후 https://계정.github.io/저장소/ 발급
  • 이후 push만 하면 자동 반영
    • 실제 사례 — 지금 이 자료가 바로 mongkoe.github.io/krema-ai-curriculum. HTML을 push → GitHub Pages가 공개. 서버 비용 0.
    적합 — HTML/CSS/JS 정적 사이트(대시보드·포트폴리오·강의자료). Next.js 같은 동적 앱은 Vercel(섹션 6)이 편하다.

    5-3GitHub CLI (gh)

    터미널에서 GitHub를 다루는 명령 도구

    gh CLI 주요 명령gh auth login # GitHub 로그인 gh repo create my-app # 저장소 생성 gh repo view --web # 브라우저로 열기 gh pr create # PR(검토 요청) 생성

    ⚡장점

    웹 클릭 없이 터미널에서 빠르게. 자동화·스크립트에 유리

    🤝Claude Code와

    Claude Code가 gh 명령으로 저장소 생성·push·배포까지 대신

    섹션 5 정리 — Git(저장)·GitHub(공유)·Pages(무료 배포)·gh(터미널). 정적 사이트는 push 한 번으로 세상에 공개된다.
    PART 2 · 배포·내부앱
    06

    GitHub → Vercel 자동 배포 파이프라인

    push만 하면 자동으로 빌드·배포되는 ‘진짜 앱’ 운영

    배포 흐름 Vercel 연동 비용

    6-1배포 흐름 + Vercel 연동

    ‘GitHub에 올리면 → Vercel이 알아서 배포’

    코드 수정→ git push→ Vercel 자동 빌드→ 라이브 반영
  • vercel.com ▸ GitHub 계정으로 로그인
  • Import ▸ 내 저장소 선택
  • 프레임워크(Next.js) 자동 감지 ▸ 환경변수(.env) 입력
  • Deploy ▸ 공개 URL 발급. 이후 push마다 자동 재배포
    • 핵심 차이 — GitHub Pages는 ‘정적 사이트’, Vercel은 Next.js 같은 동적 앱(API·서버 기능)에 강하다. 청약 대시보드(API 연동)는 Vercel이 제격.

    6-2비용 + Pages vs Vercel

    언제 무엇을, 그리고 얼마

    구분GitHub PagesVercel
    적합정적(HTML)동적(Next.js·API)
    자동 배포push 시 반영push 시 자동 빌드
    환경변수(.env)제한적지원(키 보관)
    비용무료무료(취미)~유료(상업·트래픽)
    비용 주의 — Vercel은 개인·소규모는 무료, 상업적·고트래픽은 유료 플랜. 시작은 무료로, 규모가 커지면 검토.
    섹션 6 정리 — 정적=Pages, 동적=Vercel. 둘 다 ‘push → 자동 배포’. 코드를 고치면 세상에 즉시 반영되는 운영 체계 완성.
    PART 2 · 배포·내부앱
    07

    팀원만 접근 가능한 내부앱 — Google OAuth

    ‘아무나’가 아니라 ‘우리 팀만’ 보게 하는 3가지 방법

    왜 접근 제한 방법 A 워크스페이스 방법 B Gmail 계정 방법 C 코드 레벨

    7-1왜 접근 제한 + OAuth란

    내부 데이터(고객·실적)는 ‘공개’되면 안 된다

    🔓공개 URL의 문제

    링크만 알면 누구나 접속. 고객 정보·내부 실적 대시보드엔 위험

    🔐Google OAuth

    ‘구글 로그인’으로 허용된 사람만 입장. 비밀번호 관리 불필요

    OAuth 한 줄 — “구글 계정으로 로그인하세요” → 우리가 정한 명단/조직만 통과. 안전하게 ‘내부앱’으로 운영.

    7-2접근 제한 3가지 방법

    상황에 맞게 — 조직 / 명단 / 코드

    방법기준이럴 때
    A 워크스페이스 조직회사 구글 도메인@회사.com 전 직원 허용
    B Gmail 개인 계정허용 이메일 명단특정 몇 명만(외부 협력 포함)
    C 코드 레벨 제한앱 코드에서 검사세밀한 권한·역할 분기
    선택 가이드 — 회사 도메인 있으면 A(가장 간단), 소수 지정은 B, 복잡한 권한은 C. 보통 A 또는 B로 충분.

    7-3구현 개념 + 적용

    코드는 Claude Code에 맡기고, ‘무엇을 허용할지’만 정한다

    방법 B 개념 (허용 이메일 검사)// 로그인한 사용자가 허용 명단에 있는지 확인 const allowed = ["a@company.com", "b@company.com"]; if (!allowed.includes(user.email)) { return "접근 권한이 없습니다"; }
    Claude Code 프롬프트이 대시보드에 Google 로그인(OAuth)을 붙이고, 허용된 이메일(@회사.com 도메인 또는 지정 명단)만 접근하게 만들어줘. 권한 없으면 안내 페이지로.
    섹션 7 정리 — 공개앱이 아니라 ‘내부앱’으로. Google OAuth로 우리 팀만 안전하게. 이로써 데이터→대시보드→배포→보안까지 ‘제품 운영’의 한 사이클 완성.
    WRAP-UP

    오늘 핵심 7가지

    • 공공데이터포털 — 무료·공식·실시간 데이터의 원천
    • 청약 3종 API — 분양·경쟁률·당첨을 공통 키로 연결
    • REST·보안 — URL로 호출, 인증키는 .env로 숨기기
    • Claude Code 대시보드 — Code·Design 병행, 대화로 개선
    • GitHub 배포 — commit·push, Pages로 무료 공개
    • Vercel 자동배포 — push → 자동 빌드, 동적 앱에 강함
    • Google OAuth — 팀만 접근하는 안전한 내부앱
    관통하는 원리 — “데이터(API) → 화면(Code·Design) → 공개(GitHub·Vercel) → 보안(OAuth)”. 비개발자가 ‘아이디어를 진짜 작동하는 서비스’로 만드는 전 과정. 그 증거가 이 강의 사이트다.
