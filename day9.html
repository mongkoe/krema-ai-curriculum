    3-1로그인 · 권한 허용

    발급한 인증 정보로 ‘실제 로그인’

    단순 ‘제목·발신자’가 아니라 ‘판단’을 붙여 시트로

    핵심 전환 — ‘메일 목록’이 아니라 ‘무엇부터 처리할지 판단된 표’. 이게 트리아지의 본질. 4섹션에서 이걸 대시보드로.

    PART 2 · 배포·접근제어·보너스

    GitHub → Vercel 배포

    내 컴퓨터를 넘어 인터넷으로

    배포 흐름 .env 등록 리디렉션 URI

    5-1배포 흐름 + 환경변수(.env) 등록

    로컬에서 되던 걸 ‘인터넷 주소’로 (8일차 배포 심화)

    GitHub 업로드→ Vercel이 받음→ vercel.app 주소
    핵심 — 환경변수 이전 — 로컬 .env의 인증 정보(OAuth 키 등)는 GitHub에 안 올라간다(.gitignore). 그래서 Vercel 설정에 같은 값을 직접 등록해야 배포본이 작동한다.
    Vercel에 등록할 환경변수 (예)GOOGLE_CLIENT_ID = ...apps.googleusercontent.com GOOGLE_CLIENT_SECRET = ... GOOGLE_REDIRECT_URI = https://내앱.vercel.app/api/callback SHEET_ID = ...
    순서 — ① GitHub push ② Vercel Import ③ 환경변수 등록 ④ Deploy. 변수를 빠뜨리면 ‘로컬은 되는데 배포는 안 되는’ 전형적 오류.

    5-2승인된 리디렉션 URI 추가

    배포 도메인을 구글이 ‘믿게’ 등록

  • Vercel 배포 완료 → 도메인 확인(예: my-app.vercel.app)
  • 구글 클라우드 콘솔 ▸ 웹 앱 인증정보
  • 승인된 리디렉션 URI에 배포 도메인 추가
  • 도메인 환경변수(REDIRECT_URI)도 배포 주소로 수정
    • 왜 필요 — OAuth는 ‘등록된 주소로만’ 돌아온다(보안). 배포 도메인을 리디렉션 URI에 안 넣으면 로그인 시 redirect_uri_mismatch 오류. 가장 흔한 배포 막힘.
    섹션 5 정리 — push → Vercel → 환경변수 → 리디렉션 URI. 로컬 인증이 배포 환경에서도 작동하게 잇는 게 핵심.

    외부 공개 vs 접근 제어

    누가 볼 수 있게 할 것인가 — 메일은 실데이터다

    위험 인식 보호 방법

    6-1위험 인식 + 보호 방법

    ‘URL만 알면 누구나’ 보일 수 있다는 점을 명심

    핵심 위험 — 서버가 내 토큰으로 시트를 읽어 보여주므로, 접근 제어가 없으면 URL만 알면 누구나 메일 데이터를 열람. 고객·메일은 ‘내부용 실데이터’로 취급.

    🔒방법 A — Vercel Deployment Protection

    배포에 비밀번호/팀 인증을 걸어 외부 차단
