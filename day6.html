    MCP 개념 — AI에 외부 도구를 붙이는 통신 규칙

    채팅창 안에서 노션·드라이브·웹을 직접 부리는 ‘연결 규칙’ (1-1 ~ 1-8)

    1-1 MCP란 1-2 이전 vs 이후 1-3 서버 종류 2가지 1-4 작동 구조 3단 1-5 주요 서버 5종 1-6 활용사례 4가지 1-7 커넥터 실습 1-8 장단점

    1-1MCP(Model Context Protocol)란?

    Anthropic이 만든, AI가 외부 도구에 ‘접속하는 규칙’

    한 줄 정의 — “HTTP가 웹사이트 접속 규칙이라면, MCP는 AI가 외부 도구(노션·구글드라이브·파일시스템)에 접속하는 규칙”

    🌐HTTP

    브라우저 ↔ 웹사이트를 잇는 약속. 덕분에 어떤 사이트든 접속

    🔌MCP

    AI ↔ 외부 도구를 잇는 약속. 덕분에 AI가 내 노션·드라이브를 직접 다룸

    왜 중요한가 — 표준 ‘규칙’이 생기면, 도구마다 따로 개발할 필요 없이 한 번 연결로 AI가 그 도구를 부린다. 비개발자에게도 ‘연결만 하면 되는’ 시대.

    1-2MCP 이전 vs 이후

    ‘복붙 노동’에서 ‘한 줄 명령’으로

    구분MCP 이전MCP 이후
    자료 가져오기채팅마다 일일이 복붙“노션에서 XX 찾아와서 써줘” 한 줄
    작업 흐름도구↔AI 오가며 수동AI가 도구를 직접 호출
    저장결과를 다시 복사해 저장“노션에 저장해줘”로 자동
    체감 변화 — “이 자료 좀 가져와서, 정리해서, 저기 저장해줘”를 대화 한 번에. 복붙·창 전환이 사라진다.

    1-3MCP 서버 종류 2가지

    ‘로컬’과 ‘원격’ — 어디에 설치되어 있나

    💻로컬 방식 (앱 설치형)

    • 내 PC에 설치해서 실행
    • 내 컴퓨터 파일 접근에 유리
    • 파일시스템·로컬 데이터 작업

    ☁️원격 방식 (웹 접속형)

    • 서버 주소만 알면 연결
    • 설치 없이 바로 사용
    • 노션·구글 등 클라우드 서비스
    선택 — 내 PC 파일을 다뤄야 하면 ‘로컬’, 웹 서비스를 연결하면 ‘원격’. 오늘 실습은 주로 원격(노션·캘린더) + 일부 로컬(파일).

    🤖① 클라이언트

    내가 대화하는 AI. 요청을 이해하고 도구 호출을 결정

    🌉② MCP 서버

    AI의 요청을 외부 서비스 언어로 ‘통역’하는 다리

    🗄️③ 외부 서비스

    실제 데이터가 있는 곳(노션·캘린더·네이버 등)

    핵심 — 우리가 만질 건 ‘다리(MCP 서버) 연결’뿐. 한 번 연결하면 AI가 알아서 ①↔③을 오간다.

    1-6실무 활용 사례 4가지

    MCP가 부동산 마케터의 일상을 바꾸는 장면

    📅일정 관리 자동화

    “이번 주 일정 정리하고 미팅 메일 초안 써줘”

    📊노션 자료 → 보고서

    “노션의 분양 데이터로 월간 보고서 작성”

    🏢경쟁사 분석 자동화

    “경쟁 단지 정보 조사해서 비교표로 정리”

    🕸️Apify 웹 데이터 수집

    “이 사이트 매물 정보 긁어와서 표로”

    공통점 — 모두 ‘여러 도구를 오가는’ 일이다. MCP로 도구를 붙여두면 대화 한 번이 곧 업무 처리가 된다.

    1-7실습 — 노션·구글캘린더 커넥터 연결

    Claude 데스크탑에 도구를 붙이고, 분양 문서를 요약해 노션에 저장

  • Claude 데스크탑 설치 — 데스크탑 앱(커넥터 지원)
  • 커넥터 연결 — 설정 ▸ 커넥터 ▸ 노션·구글캘린더 추가·로그인
  • 문서 요약 — “드라이브의 분양 마케팅 문서를 3줄 요약해줘”
  • 노션 저장 — “이 대화 내용을 노션 회의록 페이지로 저장해줘”
    • 실습 프롬프트구글캘린더에서 이번 주 미팅 일정을 확인하고, 드라이브의 'OO단지 분양전략.docx'를 요약해줘. 그다음 요약과 일정을 합쳐 노션 '주간 회의록' 페이지로 저장해줘.
    주의 — 커넥터 연결 시 데이터 접근 권한을 부여한다. 권한 범위를 확인하고, 민감 자료는 신중히.

    1-8MCP 장단점

    강력하지만 만능은 아니다 — 한계를 알고 써야 한다

    👍장점

    • 복붙·창 전환 제거
    • 여러 도구를 대화로 연결
    • 사람처럼 ‘판단’하며 처리
    • 연결만 하면 비개발자도 사용

    👎단점·한계

    • 한 번에 ‘하나씩’ 호출(대량 비효율)
    • 실행마다 비용·시간
    • 보안(권한·프롬프트 인젝션) 유의
    • 대량 반복은 노코드(Make)가 유리
    섹션 1 정리 — MCP = ‘AI에 도구를 붙이는 규칙’. 판단이 필요한 일에 강하다. 단, 대량 반복은 PART 2(노코드)가 맡고, 둘을 PART 3에서 결합한다.
    PART 1 · MCP 활용
    02

    MCP 멀티 커넥터 워크플로

    한 프롬프트로 AI가 여러 도구를 순서대로 호출하게 (2-1 ~ 2-5)

    2-1 워크플로 구조 2-2 작동 원리 2-3 캘린더→Gmail 2-4 웹서치→노션 2-5 웹서치→캔바 카드뉴스

    2-1멀티 커넥터 워크플로 구조

    ‘여러 도구’를 하나의 흐름으로 — AI가 순서를 알아서 짠다

    프롬프트→ AI가 필요한 도구 판단→ 도구 1 호출·결과→ 도구 2 호출·결과→ 최종 결과
    핵심 — 내가 도구를 하나씩 지정하지 않아도 된다. “무엇을 원하는지”만 말하면 AI가 어떤 도구를 어떤 순서로 쓸지 판단해 실행한다.
    전제 — Claude 데스크탑에 여러 커넥터가 미리 연결되어 있어야 한다(섹션 1-7). 연결된 도구가 많을수록 가능한 자동화가 늘어난다.

    2-2작동 원리

    ‘연결된 상태 + 한 프롬프트 = 자동 순차 호출’

  • Claude 데스크탑에 커넥터/MCP 여러 개 연결
  • 사용자가 ‘복합 작업’ 프롬프트 입력
  • AI가 작업을 단계로 쪼개고, 각 단계에 맞는 도구 선택
  • 도구를 순서대로 호출하며 결과를 다음 단계로 전달
  • 최종 산출물을 한 번에 제시
    • 비유 — 똑똑한 비서에게 “이거 조사해서, 정리해서, 저기 올려줘” 한마디 하면 알아서 여러 프로그램을 오가며 끝내는 것. 그 비서가 AI.

    2-3실습 ➊ 구글 캘린더 → Gmail 초안

    일정을 확인해 ‘일정 공유 메일’을 자동 작성 (2단계 대화)

    캘린더 MCP→ 일정 확인→ Gmail MCP→ 메일 초안
    실습 (2단계)① "이번 주 내 일정 확인해줘" ② "이 일정으로 팀에게 보낼 일정 공유 Gmail 초안 작성해줘. 제목·본문 포함, 정중한 업무 톤으로."
    핵심 — 캘린더(조회)와 Gmail(작성) 두 도구가 ‘이어서’ 동작. 일정 → 메일이 한 흐름.

    2-4실습 ➋ 웹서치로 단지 조사 → 노션 저장

    웹서치 MCP + 노션 MCP 조합 — 조사부터 정리까지

    웹서치 MCP→ 단지 정보 조사→ 노션 MCP→ 페이지 자동 저장
    실습 프롬프트'OO지역 2026년 분양 예정 단지'를 웹에서 조사해줘. 단지명·위치·세대수·예상 분양가·입주시기를 표로 정리하고, 그 결과를 노션 '분양 리서치' DB에 새 페이지로 저장해줘.
    가치 — “조사 → 표 정리 → 저장”을 한 번에. 리서치 결과가 자동으로 팀 노션에 쌓인다(지식 자산화).

    2-5실습 ➌ 웹서치 → 캔바 카드뉴스 자동 생성

    Cowork에서 캔바 커넥터 + 웹서치로 인스타 카드뉴스까지

    웹서치→ TOP 5 조사→ 캔바 커넥터→ 카드뉴스 디자인
    실습 프롬프트 (Cowork)'2026년 하반기 분양 예정 아파트 TOP 5'를 조사해서, 인스타그램 카드뉴스(5장)로 캔바에서 만들어줘. 표지 + 단지별 1장, 브랜드 톤(신뢰감 있는 네이비), 9:16.
    섹션 2 정리 — 도구를 ‘여러 개 붙이면’ 조사→정리→저장→디자인까지 대화로 연결된다. 단, ‘한 번에 하나씩’ 호출이라 대량 작업엔 한계 → PART 2 노코드가 등판.

    AI 에이전트 vs 노코드 자동화

    ‘똑똑한 비서’와 ‘자동 컨베이어 벨트’ — 언제 무엇을 쓸 것인가 (3-1 ~ 3-5)

    3-1노코드 자동화란?

    복잡한 업무 처리를 ‘코드 없이’ 자동화 — 대표 서비스 Make.com

    🧩노코드 자동화

    • 블록(모듈)을 이어 흐름을 설계
    • 한 번 설정하면 반복 자동 처리
    • 코딩 지식 불필요

    ⚙️Make.com

    • 대표 노코드 자동화 도구
    • 3,000+ 앱 연결
    • 시각적 시나리오 편집
    핵심 — MCP가 ‘그때그때 대화로’ 처리한다면, 노코드는 ‘한 번 만들어두고 계속 자동 실행’한다. 정해진 일을 24시간 반복.

    📄보고서 작성

    📅일정 관리

    알림·예약·동기화

    🔔모니터링

    부동산 적용 — 분양 문의 분류, 실거래가 수집, 인스타 발행, 월간 보고서, 상담 예약, 경쟁사 가격 모니터링… 오늘 실습으로 하나씩.

    3-3비유로 이해하기

    Cowork = 똑똑한 비서, Make = 자동 컨베이어 벨트

    🧑‍💼Claude Cowork (AI 에이전트)

    • 복잡한 판단 OK — 읽고·이해하고·결정
    • 유연하지만 ‘인건비’가 듦(비용·시간)
    • 상황마다 다른 일 처리

    🏭Make.com (노코드)

    • 빠르고 저렴 — 대량 반복에 강함
    • 정해진 일만, ‘판단’은 못 함
    • 같은 작업을 안정적으로
    핵심 통찰 — 비서에게 ‘컨베이어 벨트 작업(같은 일 1,000번)’을 시키면 비싸고 느리다. 판단이 필요하면 비서(Cowork), 반복이면 벨트(Make).

    3-45가지 핵심 차이

    항목AI 에이전트(Cowork)노코드(Make)
    자동 실행대화로 그때그때트리거로 자동·정기
    비용실행마다 비용↑오퍼레이션 단가 저렴
    실행 환경로컬·세션 의존클라우드 24시간
    에러 추적대화에 묻힘실행 로그·재시도
    보안프롬프트 인젝션 취약정해진 흐름만 실행
    보안 포인트 — AI 에이전트는 ‘읽은 내용’에 속을 수 있다(프롬프트 인젝션). 외부 콘텐츠를 그대로 처리할 땐 주의. 노코드는 흐름이 고정돼 상대적으로 안전.

    3-5상황별 선택 가이드

    둘 중 하나가 아니라 ‘작업에 맞게’

    🏭Make가 유리

    • 정해진 작업을 대량으로
    • 안정적·정기 실행
    • 비용을 아껴야 할 때
    • 예: 실거래가 매일 수집

    🧠Cowork가 유리

    • 글쓰기·해석이 필요할 때
    • 매번 다른 비정형 작업
    • 예: 경쟁 단지 전략 분석
    섹션 3 정리 — 판단=Cowork, 반복=Make. 그리고 PART 3에서 둘을 결합하면 ‘AI가 판단해서 Make 컨베이어를 돌리는’ 최강 조합이 된다.

    4-4핵심 개념 ① 시나리오와 모듈

    자동화의 ‘전체’와 ‘부품’

    🎬시나리오(Scenario)

    자동화 워크플로 전체. “문의 오면 → 분류 → 답장”이라는 하나의 흐름

    🧱모듈(Module)

    시나리오를 구성하는 각 단계(앱·동작 블록). 모듈을 이어 흐름을 만든다

    모듈1 (Gmail 수신)→ 모듈2 (AI 분류)→ 모듈3 (답장 발송)
    비유 — 시나리오 = 레시피 전체, 모듈 = 각 조리 단계. 단계(모듈)를 순서대로 이으면 요리(자동화)가 완성.

    4-5모듈 4가지 유형

    모든 모듈은 이 4가지 중 하나

    ▶️Trigger

    시작점 — “새 메일이 오면”, “폼 제출되면”

    ✏️Action

    생성·수정·삭제 — 메일 발송, 시트 추가

    🔍Search

    조회·검색 — 시트에서 고객 찾기

    🌐Universal

    범용(HTTP 등) — 모든 API 호출

    핵심 — 시나리오는 보통 Trigger(시작) → Search/Action(처리) 순. ‘무엇이 시작시키고, 무엇을 하는가’로 모듈을 읽으면 쉽다.
    Universal의 힘 — 연결 앱이 없어도 HTTP 모듈로 어떤 API든 호출 가능(섹션 5). 만능 열쇠.

    4-6핵심 개념 ② 커넥션(Connection)

    Make와 외부 서비스 사이의 ‘로그인 연결’

    🔗커넥션이란

    Gmail·시트·노션 등과 Make를 잇는 인증. 최초 1회만 로그인하면 이후 자동 연결

    🔐보안

    • Credentials에 안전 보관
    • 권한 범위 확인
    • 불필요한 연결은 삭제
    편의 — 한 번 연결해두면 그 서비스를 쓰는 모든 시나리오가 재사용. “Gmail 연결” 하나로 메일 관련 자동화 전부 가능.

    4-7핵심 개념 ③ 오퍼레이션과 크레딧

    ‘얼마나 쓰는지’를 알아야 비용을 관리한다

    ⚡오퍼레이션(Operation)

    모듈이 1번 실행될 때마다 1 오퍼레이션. 시나리오에 모듈 3개면, 1회 실행에 3 소모

    💳크레딧·요금제

    플랜별 월 오퍼레이션 한도. 초과하면 추가 구매·상위 플랜

    비용 감각(3-4 연결) — “100명에게 메일” = 100명 × 모듈 수만큼 오퍼레이션. 설계 단계에서 모듈을 줄이면 비용이 준다. 무한 루프·과다 실행 주의.
    팁 — 테스트는 ‘몇 건’으로 먼저. 전체 돌리기 전에 소량으로 검증해 오퍼레이션·오류를 확인.

    4-8간단 실습 — 고객 문의에 답장 보내기

    Trigger·Action 모듈을 직접 연결하는 첫 시나리오

  • Trigger — Gmail “Watch Emails”(새 문의 메일 감지)
  • (선택) AI — 문의 유형 분류·답변 톤 결정
  • Action — Gmail “Send Email”(답장 초안/발송)
  • 테스트 — Run once로 1건 확인 후 활성화
    • 새 문의 메일→ 유형 분류→ 답장 발송
    섹션 4 정리 — 시나리오=흐름, 모듈=단계(4유형), 커넥션=연결, 오퍼레이션=사용량. 이 4개념이면 어떤 자동화든 읽고 만들 수 있다.

    ‘만능 모듈’ HTTP의 설정값 — 이것만 알면 어떤 API든 호출

    옵션의미
    URL호출할 API 주소

    6-1핵심 패턴 5단계

    ‘수집 → (분석) → 생성 → 변환 → 발송’ — 문서 자동화의 황금 공식

    ① 폼/데이터 수집→ ② AI 분석→ ③ 템플릿 문서 생성→ ④ PDF 변환→ ⑤ 발송/저장

    📥① 수집

    폼·시트·API로 입력 받기

    🧠② (AI 분석)

    필요 시 내용 가공·개인화

    📄③~⑤ 생성·발송

    템플릿 채워 PDF로, 메일·저장

    핵심 — ②(AI 분석)는 선택. 단순 치환이면 생략, 개인 맞춤이 필요하면 AI를 끼운다.

    6-2실습 — Tally 폼 → Gemini → 맞춤 문서 → 메일

    고객이 폼을 채우면, 맞춤 문서가 PDF로 자동 발송

  • Tally 폼 — 고객이 이름·필요 서비스·예산 입력
  • Make Trigger — 폼 제출 감지
  • Gemini(AI) — 입력 기반 맞춤 문구 생성
  • 구글 문서 템플릿 복사 + 변수 치환
  • PDF 변환 → 메일 자동 발송
    • Tally 제출→ Gemini 맞춤글→ 구글문서 치환→ PDF→ 메일 발송
    결과 — 고객이 폼을 낸 ‘몇 초 뒤’ 맞춤 제안서가 메일함에. 사람 손 0. 분양 상담 신청 → 맞춤 단지 소개서 자동 발송에 직결.

    6-3활용 확장 아이디어 4종

    같은 패턴, 다른 문서 — 무궁무진

    🎓수료증 발급

    구글 슬라이드 템플릿 + 이름 치환 → 교육 수료증 자동

    🧾견적서

    구글 독스 + 항목·금액 치환 → 맞춤 견적서

    📰보도자료

    핵심 정보 입력 → AI가 보도자료 초안 자동 작성

    🏠매물 소개서

    고객 성향별 맞춤 매물 소개서 자동 생성

    부동산 핵심 — ‘고객 성향별 매물 소개서’ — 고객 정보만 넣으면 취향·예산에 맞춘 소개서가 자동으로. 영업 생산성 폭증.

    MCP × Make 결합 — 시나리오를 AI 도구로 등록

    AI의 ‘판단’ + Make의 ‘대량 처리’ = 대화 한 줄로 100명 개인화 메일 (7-1 ~ 7-5)

    7-1 MCP 단독 한계 7-2 Make 팬아웃 7-3 등록 절차 3단 7-4 1→10명 실습 7-5 개인화 일괄발송

    7-1MCP 단독의 한계

    ‘판단’은 잘하지만 ‘대량 반복’은 약하다

    ⚠️한계

    • Claude는 한 번에 하나의 도구만 호출
    • “100번 메일”을 100번 루프 → 비효율
    • 반복이 길수록 에러율↑·비용↑

    💡그래서

    판단은 AI(MCP)에게, 대량 실행은 Make에게 맡기면 된다. 각자 잘하는 일을.

    비유 — 똑똑한 비서에게 “봉투 100장 직접 접어”라고 하면 비싸고 느리다. 비서는 ‘무엇을 보낼지 판단’만, 봉투 접기(반복)는 기계(Make)에게.

    7-2Make 시나리오의 강점 — 팬아웃(Fan-out)

    하나의 입력을 ‘여러 개로 펼쳐’ 병렬 대량 처리

    단지 정보 1건→ 고객 DB 100명→ 100개 개인화 메일
    팬아웃이란 — “고객 100명 DB에 각각 맞춤 분양 소개 메일 보내줘” 한 번에, Make가 100건으로 펼쳐 병렬 처리. 대량·고속·저렴.

    ⚡대량

    수백·수천 건도 한 흐름

    ⏱️고속

    병렬 처리로 빠르게

    💰저렴

    오퍼레이션 단가로 절감

    결론 — ‘무엇을·누구에게’는 AI가 판단, ‘100번 실행’은 Make가 팬아웃. 둘을 잇는 게 7-3.

    7-3시나리오 → MCP 도구 등록 절차 3단

    Make 시나리오를 ‘AI가 부를 수 있는 도구’로 만든다

  • 인풋/아웃풋 만들기 — 시나리오에 ‘입구(받을 값)’와 ‘출구(돌려줄 값)’ 정의
  • [요청 시 실행]으로 변경 — 자동 실행 대신 ‘호출하면 실행’ 모드
  • Claude에 Make.com 연결 — Make MCP로 이 시나리오를 AI 도구로 등록
    • 시나리오→ 입구·출구 정의→ 요청 시 실행→ Claude의 도구가 됨
    의미 — 등록하면 Claude가 “이 시나리오 실행해”라고 부를 수 있다. AI가 노코드 자동화를 ‘손발’처럼 부린다.

    7-4실습 — 단지 정보 1개로 10명+ 맞춤 메일

    입구로 값을 넣고, 시나리오가 펼쳐 처리하고, 출구로 결과 회수

  • [입구/출구] 생성 — 입구: 단지 정보 / 출구: 발송 결과
  • 입구 값으로 시나리오 동작 — 고객 DB와 결합해 개인화
  • 출구로 결과 회수 — 몇 명에게 보냈는지 반환
  • AI 호출로 실행 트리거 — Claude가 시나리오를 호출
    • 실습 프롬프트 (Claude)방금 등록한 Make 시나리오를 실행해줘. 단지 정보: 'OO자이, 강남 역세권, 84㎡, 12억대, 6월 분양' 대상: 고객 DB 시트의 가망 고객 10명에게 각자 맞춤 메일 발송.
    핵심 체험 — 한 줄 대화가 ‘10명 개인화 메일 발송’으로. 이게 MCP×Make 결합의 정수.

    7-5회사메일 연동 + 개인화 메일 일괄 발송

    대화 한 줄로 ‘DB 명단 전체’에 개인화 발송

    한 줄 대화로 실행"고객 DB 시트에서 가져온 명단에게, 이 단지 정보를 각자 맞춤으로 개인화해서 일괄 발송해줘."
    한 줄 명령→ Claude 판단·호출→ Make 팬아웃→ 100명 개인화 발송
    섹션 7 정리 — MCP(판단)와 Make(대량)를 결합 = “대화 한 줄로 100명 개인화 메일”. 오늘 강의의 하이라이트이자 자동화의 완성형.
    주의 — 대량 발송은 스팸·수신 동의·발송량 제한을 지켜야 한다. 테스트는 소수로, 동의받은 명단에만.
    PART 3 · MCP × 노코드 결합
    08

    통합 활용 아이디어 + 시나리오 자동 생성

    대화 한 줄로 끝내는 3대 시나리오, 그리고 시나리오 자체를 AI가 만들기 (8-1 ~ 8-5)

    8-1 경쟁분석+리포트+메일 8-2 SNS 기획+이미지+발행 8-3 노션→보고서→메일 8-4 시나리오 JSON 자동생성 8-5 자동생성 예시

    8-1시나리오 ① 경쟁 단지 분석 + 리포트 + 메일

    조사부터 발송까지 — 한 줄 명령

    경쟁 분양가 조사→ 비교표 정리→ 메일 발송
    한 줄 명령"OO지역 경쟁 단지 분양가를 조사해서 비교표로 정리하고, 우리 팀에게 메일로 발송해줘."
    도구 조합 — 웹서치/Apify(조사) + AI(비교 정리) + Make/Gmail(발송). 섹션 5·6·7의 종합.

    8-2시나리오 ② SNS 콘텐츠 기획 + 이미지 + 발행

    기획부터 카드뉴스 생성까지 일괄

    게시물 3개 기획→ 카드뉴스 이미지 생성→ 발행/예약
    한 줄 명령"이번 주 분양 홍보 인스타 게시물 3개를 기획하고, 각각 카드뉴스 이미지까지 생성해줘. 톤은 신뢰감 있게."
    도구 조합 — AI(기획·카피) + 캔바/Nano-Banana(이미지, 3·5일차) + Make(발행 자동화). 콘텐츠 공장.

    8-3시나리오 ③ 노션 데이터 → 보고서 → 메일

    쌓인 데이터를 월간 보고서로, 그리고 발송

    노션 분양 실적→ 월간 보고서 정리→ 메일 발송
    한 줄 명령"노션의 이번 달 분양 실적 데이터를 가져와서 월간 보고서로 정리하고, 팀과 임원에게 메일로 발송해줘."
    도구 조합 — 노션 MCP(데이터) + AI(보고서 작성, PSK 구조 5일차) + Gmail(발송). 매월 반복 업무를 한 줄로.

    8-4Context7 MCP + Make MCP로 시나리오 JSON 자동 생성

    시나리오를 ‘직접 만드는’ 대신, AI에게 ‘만들어 달라’고

    📚Context7 MCP

    최신 기술 문서·레퍼런스를 AI에 제공 → 정확한 시나리오 설계

    ⚙️Make MCP

    생성된 시나리오 JSON을 Make에 등록·실행

    한 단계 위 — 이제 ‘시나리오를 손으로 만들기’조차 자동화. AI가 시나리오 설계도(JSON)를 만들고, Make에 바로 올린다. ‘자동화를 자동화’.
    의미 — 노코드보다 더 빠른 ‘노터치’. 무엇을 자동화할지만 말하면 흐름까지 AI가 설계.

    8-5자동 생성 예시 프롬프트

    말로 설명하면 시나리오 JSON이 나온다

    시나리오 자동 생성 프롬프트다음 자동화 시나리오를 Make용 JSON으로 만들어줘. "유튜브에 새 댓글이 달리면 → 구글시트에 기록 → 긍정 댓글에는 감사 대댓글 작성 → 부정 댓글은 담당자에게 알림 발송" 각 단계의 모듈(Trigger/Action)과 연결을 포함해서.
    생성 결과 (구조 예시){ "trigger": "YouTube: Watch Comments", "steps": [ { "module": "Sheets: Add Row" }, { "module": "AI: Classify sentiment" }, { "router": [ { "if": "긍정", "do": "Reply comment" }, { "if": "부정", "do": "Notify staff" } ]} ] }
    섹션 8 정리 — 3대 시나리오로 실무를 한 줄에. 나아가 시나리오 자체를 AI가 생성. ‘자동화를 만드는 일’까지 자동화되는 지점에 도달.
    WRAP-UP

    오늘 핵심 8가지

    • MCP — AI에 외부 도구를 붙이는 ‘통신 규칙’(HTTP의 AI 버전)
    • 멀티 커넥터 — 한 프롬프트로 여러 도구를 순서대로 호출
    • 에이전트 vs 노코드 — 판단=Cowork, 반복=Make
    • Make 핵심 — 시나리오·모듈(4유형)·커넥션·오퍼레이션
    • API·데이터수집 — HTTP 모듈로 공공데이터·Apify 자동 수집
    • 문서 자동화 — ‘수집→생성→변환→발송’ 한 패턴이 전부
    • MCP × Make — 시나리오를 AI 도구로 등록, 대량 팬아웃
    • 통합·자동생성 — 한 줄 명령 3대 시나리오 + 시나리오 JSON 생성
    관통하는 원리 — “판단은 AI(MCP), 반복은 자동화(Make), 둘을 결합”. 코드 없이 ‘대화 한 줄로 100명에게 개인화 메일’이 되는 게 오늘의 도달점.
